# Sprint 007: Merge Suggestions — Haiku Scanner and Admin Merge Queue

## Goal

Implement the LLM-assisted duplicate detection workflow end-to-end: run
Claude Haiku on every new user creation to score candidate duplicates, and
give administrators a merge queue to approve, reject, or defer each
suggestion.

## Use Cases Delivered

- **UC-018** — Merge suggestion generated by Haiku on user creation: collects
  candidate pairs, calls Anthropic API, creates MergeSuggestion records for
  pairs scoring >= 0.6, discards pairs below 0.6.
- **UC-019** — Admin reviews merge queue: side-by-side comparison, approve
  (merge survivors, deactivate loser), reject, or defer; atomic transaction
  with rollback on constraint violation.

## Scope

- Anthropic API client (Claude Haiku): construct the comparison prompt, parse
  confidence score and rationale from the response.
- Merge scanner service: called on user creation (the stub from Sprints 002
  and 006 is now filled in); iterates candidate pairs, calls Haiku, creates
  MergeSuggestion records at >= 0.6 threshold.
- Merge queue admin UI: list pending suggestions with confidence and
  rationale; side-by-side user comparison view.
- Approve flow: move all Logins and ExternalAccounts from non-surviving user
  to survivor; cohort assignment resolution; deactivate non-survivor;
  transactional — roll back if any constraint fails.
- Reject flow: mark suggestion rejected.
- Defer flow: mark suggestion deferred, remains in queue.
- Audit events for each suggestion created and for each approve/reject/defer
  action.
- Haiku API failure is non-fatal: user creation proceeds; failure is logged.

## Dependencies

- Sprint 001 (MergeSuggestion and AuditEvent entities).
- Sprint 002 (merge scanner call site on user creation via social login).
- Sprint 006 (merge scanner call site on user creation via Pike13 sync;
  Pike13 as the realistic duplicate source for testing).
- External: Anthropic API key.

## Non-Goals

- No auto-merge (human confirmation always required per spec).
- No batch re-scanning of existing users (scan only triggers on creation).
- No configurable threshold UI — the 0.6 threshold is a config value, not
  an admin setting.

## Rationale

Sprint 006 (Pike13 sync) is the primary duplicate source in production — a
student signs in with GitHub, then sync creates a second record from the
Pike13 person. Delivering the merge workflow immediately after Pike13 means
the two features are available together for real use. The Haiku scanner is
self-contained and has no downstream dependencies, so it fits cleanly at
this point in the sequence.
